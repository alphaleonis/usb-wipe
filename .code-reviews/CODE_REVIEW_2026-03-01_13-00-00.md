### #1 🟠 High: Browse mount leak when user presses esc during mount-in-progress

| | |
|---|---|
| **File** | `main.go:515-576` |
| **Category** | RESOURCE_LEAK |
| **Confidence** | 86/100 |
| **Found by** | design (High), broad (High) |

**Issue:** `startBrowse` sets `browseMntPath = ""` before the mount tea.Cmd runs. If the user presses esc while the mount is in-flight, `cleanupBrowse` sees empty `browseMntPath` and does nothing. The view switches to `viewDeviceDetail`. The eventual `mountResultMsg` arrives but is dispatched to `updateDeviceDetail`, which ignores it. The temp mount point is leaked (never unmounted, never cleaned up during this session).

**Fix:** Add a guard in the `mountResultMsg` handler to clean up if the view has changed:
```go
case mountResultMsg:
    if msg.err != nil {
        m.err = msg.err.Error()
        m.view = viewDeviceDetail
        return m, nil
    }
    if m.view != viewFileBrowser {
        go func() {
            runCmd("umount", msg.path)
            os.Remove(msg.path)
        }()
        return m, nil
    }
    m.browseMntPath = msg.path
    m.browseDir = msg.path
    return m, readDirCmd(msg.path)
```

### #2 🟠 High: `checkMountSafety` not re-called inside `doWipe` before destructive operations

| | |
|---|---|
| **File** | `main.go:1444` |
| **Category** | SECURITY / TOCTOU |
| **Confidence** | 87/100 |
| **Found by** | security (High), design (Medium) |

**Issue:** `checkMountSafety` runs when the user presses 'w' (line 466), but `doWipe` executes minutes later (after mode/fs/label/confirm steps). During this gap, an automounter could mount the USB device at a critical path. `doWipe` re-reads `/proc/mounts` for unmounting but does not re-check the critical-path safety list, so it would attempt to unmount critical filesystems like `/home`.

**Fix:** Call `checkMountSafety(dev)` at the top of `doWipe`, after the TOCTOU identity check but before unmounting:
```go
if err := checkMountSafety(dev); err != nil {
    return err
}
```

### #3 🟠 High: `cleanupStaleMounts` matches mount-point basename globally, not just under temp dir

| | |
|---|---|
| **File** | `main.go:1716-1732` |
| **Category** | SECURITY |
| **Confidence** | 78/100 |
| **Found by** | quick (Medium), broad (High), knowledge (SHOULD), security (Medium) |

**Issue:** `cleanupStaleMounts` checks `filepath.Base(mp)` for the `usbwipe-` prefix across ALL entries in `/proc/mounts`. Since `os.MkdirTemp` creates directories under `/tmp`, legitimate mounts are under `/tmp/usbwipe-*`. But a mount at any path whose basename starts with `usbwipe-` (e.g., `/mnt/usbwipe-backup`) would be unmounted by this root-level startup cleanup.

**Fix:** Constrain to the expected temp directory:
```go
if strings.HasPrefix(mp, os.TempDir()+"/usbwipe-") {
```
Also check that the mount source is a block device (`strings.HasPrefix(dev, "/dev/")`).

### #4 🟡 Medium: Unmount loop in `doWipe` can fail spuriously on overlapping mounts

| | |
|---|---|
| **File** | `main.go:1451` |
| **Category** | ERROR_HANDLING |
| **Confidence** | 72/100 |
| **Found by** | quick (High) |

**Issue:** The new multi-mount-point unmount loop iterates `mounts[dp]` and calls umount on each. If unmounting `mounts[dp][0]` causes `mounts[dp][1]` to also disappear (e.g., bind mounts of each other), the second umount fails. This aborts the wipe with a spurious error even though the device is fully unmounted. Regression from the prior single-mount-point code.

**Fix:** After a failed umount, re-check if the device is still mounted before returning the error:
```go
if err := runCmdStream("umount", ws, mp); err != nil {
    // Re-check — mount may have been removed by a prior umount
    fresh := parseProcMounts()
    if len(fresh[dp]) > 0 {
        return fmt.Errorf("umount %s: %w", mp, err)
    }
}
```

### #5 🟡 Medium: exFAT label length comment claims UTF-16 code units but code counts runes

| | |
|---|---|
| **File** | `main.go:849` |
| **Category** | KNOWLEDGE_PRESERVATION / BUG_LOGIC |
| **Confidence** | 78/100 |
| **Found by** | quick (Medium), knowledge (MUST) |

**Issue:** Comment says `exFAT spec: max 15 UTF-16 code units` but the code checks `utf8.RuneCountInString(val) > maxLen` which counts Unicode code points. Supplementary plane characters (emoji) are 1 rune but 2 UTF-16 code units, so a 15-emoji label would be 30 UTF-16 code units, exceeding the spec limit.

**Fix:** Either fix the comment to be accurate: `// exFAT: max 15 characters (approximate; spec measures UTF-16 code units)`, or count UTF-16 code units properly.

### #6 🟡 Medium: `checkMountSafety` uses exact string match, misses sub-paths

|---|---|

| **Category** | SECURITY |
| **Confidence** | 85/100 |
| **Found by** | security (Medium) |

**Issue:** A device mounted at `/boot/efi`, `/home/user`, `/var/lib`, or any subdirectory of a critical path bypasses the exact-match safety check.

**Fix:** Add prefix matching:
```go
for _, mp := range mounts[devPath] {

            return fmt.Errorf("refusing to wipe: %s is mounted at %s", devPath, mp)
        }
    }
}
```
