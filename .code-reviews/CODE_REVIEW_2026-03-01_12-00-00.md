### #1 🔴 Critical: Quick Verify + exFAT silently skips verification

| | |
|---|---|
| **File** | `main.go:1440-1458` |
| **Category** | DATA_LOSS / API_CONTRACT |
| **Confidence** | 94/100 |
| **Found by** | broad (Critical), design (Critical), knowledge (MUST), quick (not flagged — noted exFAT path but not as finding) |

**Issue:** When a user selects "Wipe + Quick Verify" and exFAT filesystem, the code enters the `case fsExFAT:` branch at line 1441, which runs plain `mkfs.exfat` with no bad-block check. The `wipeQuickVerify` logic (line 1447 with `-c` flag) is inside the `default:` (FAT32) branch and never executes for exFAT. The user is told they are getting "Wipe + Quick Verify" but receives a plain wipe with zero verification.

CLAUDE.md documents that quick verify for exFAT should use `badblocks -s`, but this was never implemented. The wipe mode description says "(FAT32 only)" but this is only visible during mode selection — by the time the user picks exFAT, the confirm screen shows "Wipe + Quick Verify" with no caveat.

**Fix:** Either implement the exFAT quick verify path or enforce the constraint at the UI layer:

Option A — Implement:
```go
// Before the format switch, after partition creation:
if mode == wipeQuickVerify && fstype == fsExFAT {
    emit("Running quick bad block check on %s...", part1)
    if err := runCmdStream("badblocks", ws, "-s", "-v", part1); err != nil {
        return fmt.Errorf("badblocks: %w", err)
    }
}
```

Option B — Enforce in UI: When Quick Verify is selected, skip filesystem selection and hardcode FAT32.

Also update CLAUDE.md line 45 to match whichever approach is chosen.

### #2 🔴 Critical: Unicode byte/rune mismatch in border rendering

| | |
|---|---|
| **File** | `main.go:225-229` |
| **Category** | BUG_LOGIC |
| **Confidence** | 88/100 |
| **Found by** | quick (High), broad (Critical) |

**Issue:** In `renderPane`, `hBar` is built by repeating the border character `─` (U+2500, 3 bytes in UTF-8). `titleW` comes from `lipgloss.Width()` which returns display columns, but `hBar[:1]` and `hBar[1+titleW:]` are byte-offset slices. This cuts multi-byte runes at incorrect boundaries, producing garbled border output on every render.

The condition `titleW+2 < len(hBar)` also compares display columns against byte length — always true since the byte count is 3x the column count, making the guard ineffective.

**Fix:**
```go
hBarRunes := []rune(hBar)
// ...
if titleW+2 < len(hBarRunes) {
    top = borderFg.Render(string(border.TopLeft)) +
        borderFg.Render(string(hBarRunes[:1])) +
        titleStr +
        borderFg.Render(string(hBarRunes[1+titleW:])) +
        borderFg.Render(string(border.TopRight))
}
```

### #3 🟠 High: PTY reader goroutine can outlive `cmd.Wait()`, causing interleaved output

| | |
|---|---|
| **File** | `main.go:1551-1612` |
| **Category** | CONCURRENCY |
| **Confidence** | 84/100 |
| **Found by** | quick (High), design (High) |

**Issue:** `runCmdStdinStream` calls `cmd.Wait()` at line 1612 and returns to `doWipe`, which proceeds to the next command. But the PTY reader goroutine (line 1551) may still be draining residual output. The next `runCmdStdinStream` call writes a command header to `ws.ch` while the previous goroutine is still sending output, causing interleaved lines from different commands.

**Fix:** Add a `sync.WaitGroup` or done channel. Wait for the reader goroutine to finish after `cmd.Wait()`:
```go
var wg sync.WaitGroup
wg.Add(1)
go func() {
    defer wg.Done()
    defer master.Close()
    // ... existing reader code ...
}()
err = cmd.Wait()
wg.Wait()  // ensure all output is drained
```

### #5 🟠 High: TCGETS/TCSETS return values silently ignored in `openPTY`

| | |
|---|---|
| **File** | `main.go:1501-1507` |
| **Category** | ERROR_HANDLING |
| **Confidence** | 80/100 |
| **Found by** | broad (High) |

**Issue:** The `TCGETS` and `TCSETS` ioctl calls discard the return value. If `TCGETS` fails, `attr` is zero-valued, and `TCSETS` writes garbage termios settings. Unlike the earlier ioctls in the same function (which check errno), these are fire-and-forget.

**Fix:** Check errno and return an error, matching the pattern of the preceding ioctls:
```go
if _, _, errno := syscall.Syscall(syscall.SYS_IOCTL, slave.Fd(),
    syscall.TCGETS, uintptr(unsafe.Pointer(&attr))); errno != 0 {
    master.Close(); slave.Close()
    err = fmt.Errorf("TCGETS: %v", errno)
    return
}
// ... modify attr ...
if _, _, errno := syscall.Syscall(syscall.SYS_IOCTL, slave.Fd(),
    syscall.TCSETS, uintptr(unsafe.Pointer(&attr))); errno != 0 {
    master.Close(); slave.Close()
    err = fmt.Errorf("TCSETS: %v", errno)
    return
}
```

### #6 🟠 High: Predictable `/tmp` path enables symlink attack

|---|---|
| **File** | `main.go:519` |
| **Category** | SECURITY |
| **Confidence** | 92/100 |
| **Found by** | security (High), design (Medium) |

**Issue:** Temporary mount path `/tmp/usbwipe-{partName}` is predictable. A local attacker can race to create a symlink at that path before the program, causing the root-privileged `mount` to overlay an attacker-chosen location.

### #7 🟠 High: FAT32 label allows invalid characters

| | |
|---|---|
| **File** | `main.go:858-864` |
| **Category** | INPUT_VALIDATION |
| **Confidence** | 80/100 |
| **Found by** | broad (High), security (Medium) |

**Issue:** Label validation only checks length. FAT32 labels are restricted to A-Z, 0-9, space, and limited punctuation. Unicode, control characters, and ANSI escape sequences pass through and could corrupt the filesystem label or inject terminal escape sequences into the TUI viewport via the command log.

**Fix:** Validate characters against an allowlist:
```go
upper := strings.ToUpper(val)
for _, r := range upper {
    if !((r >= 'A' && r <= 'Z') || (r >= '0' && r <= '9') || r == ' ' || r == '_' || r == '-') {
        m.err = "FAT32 label: A-Z, 0-9, space, underscore, hyphen only"
        return m, nil
    }
}
```

## Specialist Notes

### Threat Model Notes (security-reviewer)

- **Trust boundaries**: Root-privileged process interacts with sysfs (kernel), /proc/mounts (kernel), blkid (external tool), and user-selected devices. External: device names from sysfs, mount data from /proc/mounts.
- **Data sensitivity**: No PII in code, but the program destroys user data. Primary threat is irrecoverable data loss on wrong device.
- **Attack surface**: Runs as full root for entire lifetime. Creates predictable temp paths. Executes external commands with kernel-derived arguments. Complex TUI framework parses terminal input as root.
- **Privilege posture**: Entire application runs as root including TUI rendering and file browsing — only wipe/mount/umount/eject actually need root. Violates least privilege but non-trivial to split in a single-binary Go TUI.
- **Dependency posture**: Dependencies pinned with integrity hashes in go.sum. Small, well-maintained Charm ecosystem. `unsafe` import used only for PTY ioctl calls.
- **Positive**: `exec.Command` (not shell) prevents command injection. `O_NOCTTY` on PTY prevents controlling terminal takeover. `Setpgid: true` + negative PID kill correctly handles process groups.
